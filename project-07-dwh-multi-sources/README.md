# Data Warehouse with Multiple Data Sources

## О проекте

Проект посвящён развитию корпоративного хранилища данных (DWH) для сервиса доставки еды.

В существующее хранилище интегрирован новый источник данных — REST API курьерской службы. На основе объединения данных о заказах и доставках реализована аналитическая витрина для ежемесячного расчёта выплат курьерам.

---

## Бизнес-задача

Необходимо автоматизировать расчёт выплат курьерам.

Для каждого отчётного месяца рассчитываются:

- количество выполненных заказов;
- общая стоимость заказов;
- средний рейтинг курьера;
- комиссия сервиса;
- сумма выплаты за выполненные доставки;
- сумма чаевых;
- итоговое вознаграждение курьера.

Все расчёты формируются в витрине

```
cdm.dm_courier_ledger
```

которая используется бизнесом для последующего начисления выплат.

## Источники данных

### PostgreSQL

Используются существующие данные DWH:

- заказы;
- продажи товаров;
- календарные измерения.

### REST API курьерской службы

Используются три метода:

- `/restaurants`
- `/couriers`
- `/deliveries`

При работе с API реализованы:

- постраничная загрузка (paging);
- сортировка данных;
- загрузка по временному диапазону;
- обязательная авторизация через HTTP-заголовки.

---

## Архитектура DWH

### STG

Слой хранения сырых данных, полученных из REST API.

Используются таблицы:

- `stg.api_restaurants`;
- `stg.api_couriers`;
- `stg.api_deliveries`.

Особенности слоя:

- хранение полного JSON-ответа API;
- сохранение времени загрузки;
- защита от повторной загрузки через уникальные ограничения и `ON CONFLICT DO NOTHING`.

---

### DDS

Нормализованный слой хранения данных.

Используются существующие таблицы:

- `dds.dm_orders`;
- `dds.fct_product_sales`.

Добавлены новые сущности:

- `dds.dm_couriers`;
- `dds.fct_deliveries`.

На данном этапе JSON преобразуется в реляционные структуры и связывается с существующими данными DWH.

---

### CDM

Построена аналитическая витрина

```
cdm.dm_courier_ledger
```

Витрина содержит:

- отчётный период;
- курьера;
- количество заказов;
- общую стоимость заказов;
- средний рейтинг;
- комиссию сервиса;
- сумму выплаты за выполненные доставки;
- сумму чаевых;
- итоговое вознаграждение.

Все расчёты выполняются по дате заказа (`order_ts`) в соответствии с бизнес-правилами начисления выплат.

## ETL-процесс

Реализован DAG Apache Airflow, ежедневно выполняющий следующие этапы:

```
REST API
      │
      ▼
Загрузка STG
      │
      ▼
Наполнение DDS
      │
      ▼
Построение CDM
```

Последовательность задач:

1. Загрузка ресторанов.
2. Загрузка курьеров.
3. Загрузка доставок за последние 7 дней.
4. Наполнение `dds.dm_couriers`.
5. Наполнение `dds.fct_deliveries`.
6. Построение витрины `cdm.dm_courier_ledger`.

Все этапы являются идемпотентными и могут выполняться повторно без появления дубликатов данных.

- агрегирование данных;
- расчёт выплат курьерам по бизнес-правилам;
- идемпотентная загрузка данных;
- оркестрация процессов в Apache Airflow.

# Итоги

Построено расширяемое многослойное хранилище данных, объединяющее данные существующего DWH и REST API курьерской службы.

Реализована аналитическая витрина `cdm.dm_courier_ledger`, автоматически рассчитывающая ежемесячные выплаты курьерам на основании данных заказов и доставок. Витрина обновляется по расписанию и может использоваться бизнесом для формирования начислений.
